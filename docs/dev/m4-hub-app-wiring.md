# M4 hub-app wiring — the user-published `scitex_agentic_journal_hub_app`

This document describes how the user-published Django wrapper
`scitex_agentic_journal_hub_app` mounts the agentic-journal-side M4
ReReviewBadge resolver into scitex-live-paper inside scitex-hub.

**Status.** Draft. The mechanical scaffolder + submitter
(`scitex-hub app init`, `scitex-hub app submit`) ships per ADR-0002
on scitex-hub develop; this doc shows the wrapper's `_django/urls.py`
contents so an operator running `app init` knows what to drop in.
Wiring lands the moment the operator picks the B-path on the
2026-06-12 user-published-apps reframe.

## Why a wrapper

* live-paper PR #38 (merged 2026-06-13) added
  `BundleContext.re_review_badge` and a per-request
  `mount(resolver=...)` injection point.
* agentic-journal PR #32 (merged 2026-06-13) exposed
  `resolve_badge_for_paper(persistent_id, log_url=...)` returning the
  `ReReviewBadge` (or `None`).
* The 2026-06-12 reframe (PR #257 reverted via #266) forbids hub
  built-in wrappers under `apps/workspace/...` — hosts publish
  per-user Django wrappers via `scitex-hub app submit` instead.
* So the actual `mount(resolver=...)` call site lives in the
  user-published wrapper, not in scitex-hub or in agentic-journal
  proper.

## What agentic-journal ships

Two helpers under
`scitex_agentic_journal._hub_app_publisher` (see PR draft on the
agentic-journal side):

* `build_hub_resolver(load_paper, bundle_context_factory,
  bundle_source_factory, paper_state_factory,
  renderer_options_factory, hub_log_url_template)` — returns the
  `(request, paper_id, **) -> BundleContext` callable live-paper's
  `mount` expects. The wrapper hands in live-paper's
  `BundleContext` / `BundleSource.from_resolver` / `PaperState.from_db`
  / `RendererOptions` so the helper has zero live-paper imports.
* `HUB_APP_MANIFEST` — the `scitex-hub app submit` manifest payload
  (name, version, display_name, category, entry_points,
  python_requires, requires, upstream, permissions).

## Wrapper `urls.py` (drop-in template)

```python
# scitex_agentic_journal_hub_app/_django/urls.py
from django.urls import include, path

from scitex_live_paper import (
    BundleContext,
    BundleSource,
    PaperState,
    RendererOptions,
    mount,
)
from scitex_agentic_journal._hub_app_publisher import build_hub_resolver


def _load_paper(paper_id, request):
    # Host-supplied: pull the live-paper bundle for this paper.
    # The wrapper closes over the operator's storage / authz here.
    from scitex_agentic_journal_hub_app._loader import load_paper
    return load_paper(paper_id, request.user.current_project.id)


resolver = build_hub_resolver(
    load_paper=_load_paper,
    bundle_context_factory=BundleContext,
    bundle_source_factory=BundleSource.from_resolver,
    paper_state_factory=PaperState.from_db,
    renderer_options_factory=lambda: RendererOptions(embed_mode=True),
    hub_log_url_template="/aj/{paper_id}/log/",
)


urlpatterns = [
    path("apps/live-paper/<paper_id>/", include(mount(resolver=resolver))),
]
```

That's the whole wrapper-side wiring. The published app's
`apps.py` / `manifest.json` are produced by `scitex-hub app init`.

## Wrapper `manifest.json`

Generated by the scaffolder from
`scitex_agentic_journal._hub_app_publisher.HUB_APP_MANIFEST`:

```python
import json
from scitex_agentic_journal._hub_app_publisher import HUB_APP_MANIFEST

print(json.dumps(HUB_APP_MANIFEST, indent=2))
```

Carry-through fields (subject to operator ratification):

* `name`, `version`, `display_name`, `description`
* `category = "app"` → triggers `_apply_app_suffix` on submit
* `python_requires = ">=3.10"` → matches the upstream package
* `entry_points["scitex_hub.apps"]` → wrapper's `urls.urlpatterns`
* `requires` → `scitex-agentic-journal>=0.1.0a0` +
  `scitex-live-paper>=0.1.0a0`
* `upstream.package = "scitex-agentic-journal"`
* `permissions` → mirror the existing `_django/manifest.json`
  permission list

## Boundary callouts

* **No hub edits.** The wrapper is a separate package submitted via
  `scitex-hub app submit`; scitex-hub does not gain any
  `apps/workspace/live_paper_app/` directory.
* **No live-paper edits.** PR #38 is the host-injection contract;
  this wrapper is the host that performs the injection.
* **No agentic-journal-side Django coupling.**
  `_hub_app_publisher/_resolver_adapter.py` has zero live-paper
  imports — the wrapper passes the live-paper types in as
  parameters. Tests for the adapter can use simple fakes.
